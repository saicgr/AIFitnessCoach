## Phase 0 — Canonicalize the exercise vocabulary

**Why first:** every Sonnet sub-agent must be **constrained** to canonical exercise names. If the prompt allows free-form names, we recreate the 51.7% miss rate.

### 0.1 Resolve the canonical map buckets

`program_exercise_name_map` (regular table) is now built against the **blessed subset only** (`image_url` + `video_url` populated) by `backend/scripts/build_canonical_map.py` using a **stacked-strategy pipeline** (2026-05-09 rebuild). First strategy that succeeds wins:

| Strategy | Tier | Confidence | Rule |
|---|---|---|---|
| 1 — Plural / morphological normalization | **A0** | 100 | After singularization + parenthetical-as-equipment + handstand collapse, raw normalized == blessed normalized. Also covers a `CANONICAL_DEFAULTS` alias map for plain ambiguous names (`plank → Plank On Elbows`, `pull-up → Pull-Up Normal Grip`, `face pull → Cable Face Pull With Rope`, `leg press → Leg Press Machine Normal Stance`, etc.) |
| 2 — Token-set bag equality | **A1** | 95 | After normalize+singularize, raw_tokens == blessed_tokens (ignoring stop-words) |
| 3 — Composite multi-signal scorer | **B** | 50–92 | composite = 0.40·trigram + 0.25·token_jaccard + 0.20·movement_phrase + 0.15·body_region − 0.50·equipment_explicit_mismatch − niche_penalty − extra_token_penalty. Accepted iff composite ≥ 0.50 AND no explicit equipment mismatch. |
| 4 — Same-equipment fallback | **C** | 36–45 | Raw lacks equipment → adopt the most common blessed variant of that movement; flagged `verify_equipment_default`. |
| 5 — UNMAPPED | UNMAPPED | 0–50 | best_composite × 100; suggested_action ∈ {needs_media_production, swap_in_prompt, add_alias, typo_or_ambiguous, banned}. |

Movement phrases are matched longest-first (e.g. "bench press" before bare "press"). The `equipment_explicit_mismatch` penalty (−0.50) is the hard guard against the previous failure mode where `pull-up` matched `Band Pull-Up`, `face pull` matched `Band Face Pull`, etc.

Run snapshot (2026-05-09, 8,708 distinct raw program-exercise names against 1,859 blessed entries):

| Tier | Count | % of raw names | Treatment |
|---|---|---|---|
| A0 | **277** | 3.2% | Exact match after normalize+singularize, or canonical-default alias hit. |
| A1 | **150** | 1.7% | Token-set bag equality. |
| B | **2,062** | 23.7% | Composite ≥ 0.50, no explicit equipment mismatch. |
| C | **297** | 3.4% | Same-equipment fallback (verify_equipment_default). |
| UNMAPPED | **5,922** | 68.0% | Best candidate < 0.50 or explicit equipment mismatch. See `docs/library_gaps_for_media_production.csv` (1,284 rows) for the prioritized media-production list. |

Confidence distribution across all accepted rows: >90: **428** | 70–90: **614** | 50–70: **1,447** | <50: **297**.

Two human-verifiable artifacts are produced on every run:

- `docs/EXERCISE_NAME_MAPPING.md` — every accepted (A/B/C) mapping, sorted alphabetically by raw name, with composite + per-signal ✓ columns + `program_count`.
- `docs/EXERCISE_NAME_UNMAPPED.md` — every UNMAPPED raw name, ranked by `program_count`, with the nearest blessed candidate, the reason it failed (`composite=…, movement=…, equipment=…, body=…`), and a `suggested_action` ∈ {`swap_in_prompt`, `needs_media_production`, `typo`, `banned`, `needs_review`}. A second section groups the same entries by suggested action so reviewers can scan e.g. all `needs_media_production` items together.

Plus the machine-readable `docs/exercise_name_mapping.csv` (one row per raw name including UNMAPPED, with `raw_normalized` for re-running).

## Phase 1 — Build the generation queue

`backend/scripts/build_generation_queue.py` (to be created) generates `docs/generation_queue.jsonl`:

```json
{"job_id": "sha256(program_id+intensity+duration)",
 "program_id": "uuid", "program_name": "...", "category": "fat_loss",
 "split_type": "push_pull_legs", "intensity_level": "Hard",
 "duration_weeks": 12, "sessions_per_week": 4,
 "priority": 10, "status": "pending", "claim_token": null}
```

Priority ordering:
| Priority | What |
|---|---|
| 1 | The 6 UNDONE programs from `programs_to_regenerate.csv` (54 cells) |
| 2 | Medium intensity missing durations (1,572 cells) — fastest match-rate win |
| 3 | Easy × 4/8/12 wk for catalog (~4,300 cells) |
| 4 | Hard × 4/8/12 wk for catalog (~4,300 cells) |
| 5 | Easy × 2/3/6 wk + Hard × 2/3/6 wk (~6,500 cells) |

State table for idempotent claim/release:

```sql
CREATE TABLE program_variant_generation_queue (
    job_id TEXT PRIMARY KEY,
    program_id UUID NOT NULL,
    intensity_level TEXT NOT NULL,
    duration_weeks INT NOT NULL,
    priority INT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',  -- pending|claimed|done|failed|quarantined
    claimed_by TEXT, claimed_at TIMESTAMPTZ,
    attempts INT DEFAULT 0,
    last_error TEXT,
    UNIQUE(program_id, intensity_level, duration_weeks)
);
CREATE INDEX ON program_variant_generation_queue (status, priority);
```

Atomic claim with `FOR UPDATE SKIP LOCKED` so multiple coordinator sessions don't double-claim:

```sql
WITH next AS (
  SELECT job_id FROM program_variant_generation_queue
  WHERE status = 'pending' ORDER BY priority, job_id
  LIMIT $1 FOR UPDATE SKIP LOCKED
)
UPDATE program_variant_generation_queue q SET
  status='claimed', claimed_by=$2, claimed_at=now(), attempts=attempts+1
FROM next WHERE q.job_id=next.job_id RETURNING q.*;
```

## Phase 2 — Sonnet swarm execution

### Two coexisting deployment shapes

| Shape | When | How |
|---|---|---|
| **A) Single Claude Code session, fan-out per wave** | Interactive runs, debugging, small batches | One coordinator runs in your CC session and spawns N sub-agents per wave via the Agent tool; coordinator validates + writes after each wave |
| **B) Headless `claude -p` workers** | Long unattended runs, true parallelism across many cores | Python orchestrator launches N `claude --print --output-format=stream-json` subprocesses; each consumes one batch from the queue and exits; orchestrator restarts as workers finish |

Shape A is simpler; shape B scales further. Use A first to validate the pipeline on Priority 1+2 (~1,626 cells), then promote to shape B for the long tail.

### 2.1 Wave size + batching

| Setting | Value | Rationale |
|---|---|---|
| Sub-agents per wave (shape A) | 8 | Comfortable for parent context; fits Sonnet rate limits |
| Concurrent workers (shape B) | 16 | Pro Max API parallelism comfort zone |
| Cells per sub-agent | 6 | A cell = 1 (program × intensity × duration). One sub-agent generates 6 variants in ~6–10 min |
| Wave throughput (A) | 48 cells / ~10 min | ≈ 290 cells/hr |
| Throughput (B) | 96 cells / ~10 min | ≈ 580 cells/hr |
| Time to clear 16,820 cells | ~58 hr (A) or ~29 hr (B) | Spread across multiple sessions / a weekend |

### 2.2 The sub-agent prompt template (the load-bearing piece)

```
You are generating workout-program variant data for Zealova/FitWiz.

CONSTRAINTS — violating any constraint causes the job to be rejected:
1. Use ONLY exercise names from the blessed vocabulary attached below
   (exact-match, case-insensitive). No new names. No abbreviations.
2. Each exercise object MUST include `exercise_library_id` populated from
   the vocabulary's `id` field. `in_library` MUST be `true`.
3. JSON output MUST conform to the schema in
   `docs/program_variant_week.schema.json`.
4. Sets 1–8, reps 1–50 (or seconds 5–300 for time-based), rest 30–300s,
   primary_muscles ∈ {muscle vocabulary}, equipment ∈ {equipment vocabulary}.
5. Number of weeks MUST equal `duration_weeks`. Workouts/week MUST equal
   `sessions_per_week`. No duplicate workouts within a week.
6. Apply progressive overload week-over-week (volume ↑ or load ↑ or
   density ↑). Last week of program is a deload (intensity ↓ ~30%).
7. Intensity dial:
   - Easy   → ~70% RPE, longer rests, more bilateral / fundamental moves
   - Medium → ~80% RPE, baseline
   - Hard   → ~90% RPE, more advanced exercises (must still be in vocab),
              shorter rests, tempo / paused / unilateral / cluster sets

INPUT:
  - program metadata (name, category, split_type, sessions_per_week, goal)
  - intensity_level
  - duration_weeks
  - blessed_vocabulary (sliced to exercises matching program's equipment +
    primary muscle groups; ~150–400 names per call)

OUTPUT (one JSON object per cell, line-delimited):
{
  "job_id": "...",
  "variant": {
    "intensity_level": "Hard",
    "duration_weeks": 12,
    "phase": "hypertrophy",
    "focus": "chest_back"
  },
  "weeks": [
    { "week_number": 1, "phase": "accumulation",
      "workouts": [ {... full workout schema ...} ] },
    ...
  ]
}

Do NOT include any prose. Do NOT include exercises whose name does not
appear in the attached vocabulary list verbatim.
```

### 2.3 Per-cell validation gates (before staging)

`backend/scripts/validate_generated_variant.py` runs synchronously per returned object:

1. JSON parses cleanly
2. `len(weeks) == duration_weeks`; week numbers are 1..N with no gaps
3. Each week has exactly `sessions_per_week` workouts
4. For every exercise across every workout:
   - `exercise.name.lower().strip()` exists in blessed vocabulary
   - `exercise_library_id` matches the vocabulary's id for that name
   - `in_library` is `true`
   - sets/reps/rest/duration within sane ranges
5. No duplicate workout objects within a week
6. Progressive overload signal: any of (avg_volume, avg_load, avg_density) on week N is ≥ week N-1 (except deload)

Failures return the cell to the queue with `attempts++`. After 3 failed attempts the cell goes to `status='quarantined'` for human triage.

### 2.4 Row-by-row direct insert (no staging tables)

**Key principle: every validated cell is committed to `program_variants` + `program_variant_weeks` immediately, in its own transaction. No staging step. No batch promotion.**

This makes the entire run trivially resumable — if the swarm crashes, the laptop closes, the Pro Max cap hits, or the network drops mid-wave, every variant generated up to that moment is already durably in Supabase. The next session reads `program_variant_generation_queue.status` and skips done cells with zero replay cost.

Two small additions to `program_variants` (idempotency + traceability — additive, no breaking change):

```sql
ALTER TABLE program_variants
  ADD COLUMN IF NOT EXISTS job_id TEXT UNIQUE,            -- sha256(program_id+intensity+duration)
  ADD COLUMN IF NOT EXISTS generated_by TEXT,             -- 'sonnet-4-6' for this round
  ADD COLUMN IF NOT EXISTS generated_at TIMESTAMPTZ;
CREATE INDEX IF NOT EXISTS idx_program_variants_generated_by ON program_variants(generated_by);
```

Per-cell write path (single transaction, all-or-nothing):

```sql
BEGIN;

-- 1. Insert the variant. ON CONFLICT (job_id) DO NOTHING makes retries safe.
INSERT INTO program_variants (
    id, base_program_id, intensity_level, duration_weeks, variant_name,
    phase, focus, generation_model, generation_tokens, generation_cost_usd,
    job_id, generated_by, generated_at
) VALUES (
    gen_random_uuid(), $program_id, $intensity, $duration, $variant_name,
    $phase, $focus, 'sonnet-4-6', $tokens, 0,
    $job_id, 'sonnet-4-6', now()
)
ON CONFLICT (job_id) DO NOTHING
RETURNING id;

-- 2. Insert the N week rows for that variant.
INSERT INTO program_variant_weeks (variant_id, week_number, phase, focus, workouts)
SELECT $variant_id, week_no, week_phase, week_focus, workout_jsonb
FROM unnest(...) AS w(week_no, week_phase, week_focus, workout_jsonb);

-- 3. Mark the queue cell done in the SAME transaction.
UPDATE program_variant_generation_queue
   SET status='done', claimed_by=NULL, claimed_at=NULL
 WHERE job_id = $job_id;

COMMIT;
```

If any step fails the transaction rolls back, leaving the queue cell `claimed` (auto-released after 30 min) for retry. **No partial writes are ever visible.**

### 2.5 Resume semantics

Every coordinator session opens with:

```sql
-- Auto-release stale claims (crashed sessions)
UPDATE program_variant_generation_queue
   SET status='pending', claimed_by=NULL, claimed_at=NULL
 WHERE status='claimed' AND claimed_at < now() - interval '30 minutes';
```

Then drains the queue using the `FOR UPDATE SKIP LOCKED` claim from §1. Because both validation AND insert AND queue-update happen in one transaction per cell, **the only durable state is "pending / claimed / done / quarantined" in the queue, and the live tables are always self-consistent.**

Spot-QA happens out-of-band by querying the live tables filtered by `generated_by='sonnet-4-6'`. If a batch turns out bad, rollback is one statement:

```sql
DELETE FROM program_variants
 WHERE generated_by='sonnet-4-6'
   AND generated_at BETWEEN $start AND $end;
-- CASCADE removes the program_variant_weeks rows.
-- Queue cells remain status='done'; bulk reset before re-running:
UPDATE program_variant_generation_queue
   SET status='pending', attempts=0
 WHERE job_id IN (SELECT job_id FROM ...);
```

Every 1,000 done cells, re-run:




## Risk register

| Risk | Mitigation |
|---|---|
| Sub-agent ignores "use only blessed vocab" rule | Per-cell validation gate rejects + retries; 3 strikes → quarantine |
| Pro Max usage cap hits mid-run | Queue is resume-safe; pick up next session |
| Coordinator crashes mid-wave | Per-cell transaction; `FOR UPDATE SKIP LOCKED` claim + 30-min auto-release of stale claims; nothing partial ever lands |
| Pro Max cap mid-run | Every committed cell is durable; next session reads queue and resumes from the next `pending` row — no replay needed |
| Generated programming is unsafe / nonsensical | Pre-write validator (sets/reps/rest sanity, vocab match) + injury-safety check from `services/workout_safety_validator.py` |
| Catastrophic bad batch lands in live | `generated_by='sonnet-4-6'` tag enables one-line rollback: `DELETE FROM program_variants WHERE generated_by='sonnet-4-6' AND generated_at > '...'` (CASCADE drops the weeks) |
