📝 POST TEXT (copy this):
A user opens their workout, taps into an exercise called Ski Erg Easy, and a completely different exercise loads.

I found that bug live in production, not in a demo or a test run.

I'm building an AI fitness coach solo. The schedule screen shows a session's exercise name from one path, then resolves what actually opens through a separate path, name to alias to canonical exercise to media. Those two paths are supposed to agree. This time they did not.

The root cause was more interesting than the bug itself. The system that decides what exercise actually opens is a fuzzy matcher, built to find some real exercise with working video, not necessarily the one whose name is on screen. It is designed to be forgiving. Forgiving systems are exactly the ones that drift quietly, because they never throw an error. They just serve you something slightly wrong and keep going.

I wrote an audit script that walks every session in the catalog, compares the displayed name against what it actually resolves to, and flags every disagreement. I made a deliberate call not to auto fix any of it in bulk. The same fuzzy matching that caused the bug could just as easily paper over it with a wrong answer. Every fix gets reviewed by name before it ships.

Then I wired the program builder to warn, non fatally, the moment a newly authored exercise's name diverges from its own resolution. It catches the next version of this before publish, not after a user finds it.

None of this is a feature. Nobody sees a new button. But the entire premise of Zealova is that the plan on your screen is the plan you are actually doing. That trust breaks the third time a name and the exercise underneath it are two different things.

If you are building anything that resolves data through a chain of lookups, I would bet you have a version of this bug sitting quietly in your system right now.

What is the most forgiving part of your own system, and have you actually checked what it is quietly getting wrong?
