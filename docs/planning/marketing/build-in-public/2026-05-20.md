📝 POST TEXT (copy this):
I shipped a bug last month that I only found today.

Every user running Zealova in Chicago, Los Angeles, or Denver was getting the wrong workout served to them every morning. The app said chest day. The actual workout for today was something else entirely.

No crash. No error thrown. No user report. Just silent, confident wrongness.

The cause was one assumption repeated in six different backend files: that today means midnight UTC. In Central time, midnight UTC is 7pm the night before. So every workout written to the database at midnight UTC was technically anchored to yesterday's calendar date in local time. The query that fetched today's workout read today the same UTC way, found the midnight-UTC workout for tomorrow, and returned it.

The phone already sent the user's IANA timezone on every request. The backend was ignoring it.

The fix was conceptually simple and mechanically tedious: anchor every scheduled_date write at noon local time instead of midnight UTC. Plumb the IANA header through every endpoint that touches a date. Add a self-healing write-through so the database's timezone column stays fresh even from cron jobs that never see a live request. Then write 59 assertions covering DST transitions in both hemispheres and every UTC offset from -11 to +14. Add a CI lint gate that flags any new midnight-UTC write before it merges.

Three commits. Most of the day.

The lesson I keep relearning: the bugs that matter most are not the ones that crash. They are the ones that return the wrong answer confidently and silently. A crash tells you something is broken. A wrong answer requires you to know what the right answer should have been.

What is the most expensive silent bug you have shipped?
