## Tables Used

Every row in this report is derived from one of the tables / views below. The script reads them live from Supabase via `DATABASE_URL` in `backend/.env`.

- `branded_programs` — name, category, split_type, duration_weeks, program_metadata. One row per program.
- `program_variants` — variant counts, intensity_level, duration_weeks, generation_cost_usd. Joined to programs via `base_program_id`.
- `program_variant_weeks` — week-level workout JSONB; the `exercises[]` array inside `workouts` feeds the exercise-name extraction and the match-rate calc.
- `exercise_library_cleaned` (materialized view) — canonical exercise names, used as the Tier-1 lookup target. The **blessed predicate** (`image_url IS NOT NULL AND video_url IS NOT NULL`) restricts this to the subset of exercises that have both an illustration and a vertical demo video; this subset is the only vocabulary the program generator may emit.
- `exercise_aliases` — `alias_name` → `canonical_exercise_id` map, merged into the Tier-1 lookup.
- `program_exercise_name_map` (regular table, populated by `backend/scripts/build_canonical_map.py`) — 4-tier robust 1:1 map against the blessed subset only (`0.5*trigram + 0.3*token_jaccard + 0.2*movement_match` plus movement / equipment / body-region corroboration). Tiers: A (auto-accept), B (composite ≥ 0.65 + 2-of-3 signals), C (composite 0.50–0.65 + all 3 signals), UNMAPPED. Rows in tier ∈ {A, B, C} drive the **mediated** match rate.
- `docs/programs_to_regenerate.csv` (file, **NEW**) — the 6 deleted broken programs are re-injected as `status=UNDONE` rows so the resume plan stays visible in the report.

## How to use the CSVs

### `programs_validation.csv` (one row per program)

- Filter `data_validity_flag = BROKEN` for programs that need to be regenerated from scratch (no variants or no week data).
- Filter `data_validity_flag = PARTIAL` for programs that have variants but are missing intensities/durations or have a low library-match rate.
- `match_rate_pct` < 50% indicates the variant generator is inventing exercise names that don't exist in `exercise_library_cleaned` — fix by canonicalizing those names (see next section) before the next generation run.
- `exercise_library_id_populated` is the count of exercise records in JSONB that already have a non-null `exercise_library_id`. If this is 0 or near-0 even when `match_rate_pct` is high, the post-generation linker step never ran and should be back-filled.
- `est_cost_to_complete_usd` uses the placeholder pricing constants above. Multiply by your real input/output prices once confirmed.

### `programs_unmatched_exercises.csv` (canonicalization gap)

One row per distinct exercise name that did NOT match the library, with the number of programs that reference it. Workflow:

1. Sort by `program_count` desc — the top of the file is the highest-leverage canonicalization work.
2. For each row, decide whether the exercise should:
   - Map to an existing library entry → add to `exercise_aliases` (`alias_name` → `canonical_exercise_id`).
   - Be added to the library → insert into `exercise_library_cleaned` (then `SELECT refresh_exercise_library_cleaned();`).
   - Be banned/replaced in prompts → update the variant generator's allowed-exercise list.
3. Re-run this validation script after each batch of fixes to watch the global match rate climb.

The normalization used here lowercases, strips punctuation, collapses whitespace, expands token aliases (bb→barbell, db→dumbbell, kb→kettlebell, ohp→overhead press, rdl→romanian deadlift, bw→bodyweight, pushup→push up, pullup→pull up, …). If a real exercise still misses, extend `_TOKEN_ALIASES` in `validate_programs.py` rather than mutating the data.
